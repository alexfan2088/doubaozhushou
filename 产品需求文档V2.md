# 豆包助手 V2.0 产品需求文档

## 1. 文档信息

- 产品名称：豆包助手 V2.0
- 文档版本：V2.0 Draft 4（V1/V2 单界面模式切换）
- 目标平台：Android 16，首批适配 vivo 手机
- 外接设备：Newline NP-S2201、支持双向通话的蓝牙设备
- 核心服务：火山引擎豆包端到端实时语音大模型
- 文档日期：2026-06-30
- 目标规模：10 万至 100 万注册用户

## 1.1 实施原则

V2 的长期设计容量为 100 万注册用户，但项目从单机和小规模用户开始，不在技术验证阶段一次性建设完整的大规模平台。

实施遵循以下原则：

- 先验证核心价值，再建设规模化能力。
- 当前阶段只实现下一阶段确实需要的功能。
- 客户端从第一天保留清晰接口，避免以后推倒重写。
- 服务端按用户规模和运营指标逐步拆分，不提前引入不必要的分布式组件。
- 每个阶段设置明确退出条件，达到指标后再进入下一阶段。
- 永久密钥安全、用户隐私和费用上限不因规模小而降低标准。

## 2. 背景

V1 通过启动豆包 App、使用无障碍服务点击“实时语音通话”，并辅助选择 Type-C 或蓝牙音频设备，实现外接会议终端上的连续语音对话。

该方案存在根本限制：

- 豆包 App 的实时语音能力每天只有约 60 分钟可用时间。
- 额度可能按实时语音功能处于开启状态的时长消耗，而不是按实际交流时长消耗。
- 豆包 App 页面、控件和无障碍节点升级后，自动化流程可能失效。
- 本应用无法完整控制豆包 App 的录音、播放、会话状态、错误恢复和使用成本。
- 用户关闭实时语音页面后，错误的自动点击状态可能再次进入实时语音。

V2 将不再启动、操作或依赖豆包 App，而是在自己的 Android App 中直接接入火山引擎提供的豆包端到端实时语音服务，以付费 API 形式实现可控的连续语音能力。

## 3. 产品目标

### 3.1 核心目标

- 完全移除对豆包 App 的运行依赖。
- 在本应用内完成麦克风采集、云端实时对话和扬声器播放。
- 支持 Type-C 和蓝牙双向音频设备，Type-C 优先。
- 支持自然多轮对话、用户打断和会话恢复。
- 将会话时长、费用、开始与结束状态置于本应用控制之下。
- 保持适老化使用方式，首次配置后日常操作尽可能简单。

### 3.2 成功标准

- 未安装豆包 App 时，V2 仍能正常完成实时语音对话。
- 不受豆包 App 每日 60 分钟额度约束。
- 连续完成 30 轮有效问答，无需每轮点击录音按钮。
- Type-C 或蓝牙断开后立即停止云端音频上传和播放。
- 用户能够明确开始、暂停和结束会话。
- 应用能够展示本次时长、当日用量和估算费用。
- 产品能够从 10 万注册用户扩展至 100 万注册用户，不因共享凭证、单点服务或固定并发配置产生系统性风险。
- 高峰并发达到已采购容量的 80% 前能够预警、扩容或执行明确的准入策略。

### 3.3 当前阶段目标

当前只推进 P0 和 P1：

- P0 完成 1 台目标手机、Newline 和蓝牙设备的技术闭环。
- P1 支持 10～100 名受控测试用户。
- 暂不建设跨地域容灾、复杂计费平台和大型运营后台。
- 代码必须保留语音客户端、凭证、会话和用量模块边界，为后续扩展做好准备。

### 3.4 V1 与 V2 共存原则

- 产品保持一个 Android App、一个桌面图标和一个主入口。
- 主界面提供 V1/V2 模式选择。
- 新安装和未保存选择的用户默认进入 V2。
- 用户选择 V1 时，显示已经封板的 V1 原有界面，并使用原有豆包 App 自动化流程。
- 用户选择 V2 时，显示 V2 界面，并使用本应用内的豆包 API 实时语音流程。
- 用户选择在应用和手机重启后保留。
- V1 已经封板，除阻断性安全问题或目标系统完全不可用外，不再增加功能或修改交互。
- V1 和 V2 不得同时建立语音会话或同时占用麦克风。

## 4. 产品范围

### 4.1 V2.0 范围内

- 火山引擎账号、语音应用和实时语音模型接入。
- 本应用内的实时双向语音会话。
- Type-C USB Audio Class 输入输出。
- 支持 HFP/SCO 或 BLE Audio 通话能力的蓝牙设备。
- 音频设备检测、选择、优先级和断线恢复。
- 开始、暂停、继续、结束会话。
- 欢迎词、人设、音色和基本对话参数。
- 连接、监听、思考、回答、错误等状态展示。
- 本次会话时长、当日累计时长和费用估算。
- API 凭证安全方案。
- 必要的日志和家属诊断能力。
- 用户账号、设备绑定、订阅权益和服务准入。
- 多租户额度、并发和成本控制。
- 分批发布、远程配置和紧急停服开关。
- 面向运营和客服的管理后台基础能力。

### 4.2 V2.0 范围外

- 继续控制豆包 App 的实时语音页面。
- 通过无障碍服务模拟点击豆包控件。
- 绕过火山引擎计费、配额或服务限制。
- 首版支持所有 Android 厂商和全部蓝牙协议。
- 首版实现自研大模型、ASR 或 TTS。
- 默认保存用户的原始语音。
- 无明确授权的声音克隆。

## 5. 目标用户

### 5.1 老年用户

- 不熟悉 Android 设置和复杂操作。
- 希望直接通过会议终端与 AI 交流。
- 需要大字、明确状态、少操作和语音反馈。

### 5.2 家属或管理员

- 完成首次安装、权限和 API 配置。
- 选择音频设备、音色和人设。
- 查看用量、费用、网络和设备状态。
- 处理授权、欠费、配额和异常。

## 6. 核心使用场景

### 6.0 模式选择

1. 用户打开豆包助手。
2. 页面顶部显示 V1/V2 模式选择控件。
3. 首次安装默认选中 V2。
4. 选中 V1 后，页面主体切换为封板的 V1 界面。
5. 选中 V2 后，页面主体切换为 V2 界面。
6. 切换前若当前模式存在活动会话，应用先明确结束当前会话并释放麦克风、扬声器和网络连接。
7. 切换完成后只初始化新模式需要的模块。

模式选择文案应让普通用户可以理解：

- V1：豆包 App 模式（兼容版）
- V2：豆包 API 模式（推荐）

V2 旁显示“默认”或“推荐”标识；V1 旁显示“兼容模式”标识。

### 6.1 Type-C 对话

1. 用户连接 Android 手机和 Newline NP-S2201。
2. 应用确认 USB 麦克风和扬声器均可用。
3. 应用将 Type-C 设为当前会话音频设备。
4. 用户点击“开始对话”，或在家属配置允许时自动开始。
5. 应用建立豆包实时语音会话。
6. 用户通过 Newline 麦克风说话。
7. 豆包回复从 Newline 扬声器播放。
8. 用户可在回复过程中打断并继续提问。
9. 线缆断开后，应用立即结束或暂停会话。

### 6.2 蓝牙对话

1. Type-C 未连接。
2. 应用检测已选择的蓝牙通话设备。
3. 蓝牙未开启时，由系统授权界面请求开启。
4. 蓝牙设备连接后成为当前输入输出设备。
5. 用户开始实时语音会话。
6. Type-C 后续接入时，应用提示并按配置切换到 Type-C。

### 6.3 无设备状态

- Type-C 和蓝牙均未连接时显示“等待连接”。
- 不采集麦克风。
- 不建立云端会话。
- 不产生实时语音费用。
- 不启动豆包 App，也不执行任何无障碍点击。

### 6.4 长时间静默

- 会话开始后，应用保持短期连续监听。
- 达到可配置的静默时长后，进入低费用等待状态。
- V2.0 首版可要求用户点击继续。
- 后续版本可使用本地 VAD 或唤醒词恢复云端会话。

## 7. 功能需求

### 7.1 账号与服务配置

- 支持配置服务环境、应用标识、模型、音色和用户标识。
- 正式环境不得在 APK 中保存永久主账号 AK/SK。
- 应显示服务是否开通、余额或配额是否可用。
- 配置失败时给出明确原因，不以“未知错误”代替服务端错误。
- 每个最终用户必须有独立业务用户 ID，不得让所有终端共用同一个不可追踪的 `uid`。
- 每台设备应有独立设备 ID，并与用户账号建立可撤销的绑定关系。
- 账号、设备、订阅权益和语音会话必须能够关联审计。

### 7.2 实时语音会话

- 支持建立、维持和结束端到端实时语音会话。
- 支持连续多轮交流。
- 支持服务端自动判停。
- 支持用户在模型回答时打断。
- 支持配置人设、说话风格和欢迎词。
- 支持选择公开可用音色。
- 服务端允许时支持联网和外部知识能力。
- 单次云端连接达到服务限制前应主动重连，并尽量保持用户无感。

### 7.3 音频采集

- 仅在明确的对话状态中申请麦克风权限并采集音频。
- 优先采集 Type-C 麦克风；Type-C 不可用时使用已选择的蓝牙通话麦克风。
- 音频格式、采样率、位深和声道数应符合实时语音 API 或 SDK 要求。
- 采集链路不得持续丢帧、重复送帧或无限缓存。
- 应记录丢帧、缓冲延迟和输入电平等诊断指标。

### 7.4 音频播放

- 模型返回的音频必须播放到当前选中的 Type-C 或蓝牙设备。
- 支持边接收边播放。
- 支持收到用户打断事件后快速停止未播放的模型音频。
- 防止旧会话残留音频在新会话中播放。
- 音量可调，且切换设备后保持合理音量。

### 7.5 音频路由

- 优先级固定为 Type-C > 已选择蓝牙设备 > 无设备。
- Type-C 输入输出必须同时存在，才可判定完整双向音频就绪。
- 蓝牙设备必须具备通话输入和输出能力，不能只支持 A2DP 播放。
- 路由变化时应暂停音频流，完成切换后再恢复。
- 断开设备后不得继续使用失效的 `AudioDeviceInfo`。

### 7.6 会话控制

- 提供“开始对话”“暂停”“继续”“结束”入口。
- 明确显示以下状态：
  - 等待连接
  - 设备已连接
  - 正在连接云端
  - 正在听
  - 用户正在说话
  - 豆包正在回答
  - 已暂停
  - 网络异常
  - 服务不可用
- 重复点击开始不得创建多个并行会话。
- 用户主动结束后，不得自动重新开始，除非用户再次操作或启用了明确的自动开始配置。

### 7.7 欢迎词

- 欢迎词可由家属在界面中编辑。
- 默认欢迎词为“豆包豆包，你好啊”。
- 欢迎词配置在应用和手机重启后保留。
- V2 欢迎词应优先通过实时语音模型产生，不再依赖 Android 系统 TTS。
- 欢迎词只能在会话成功建立后播放一次。

### 7.8 用量与费用

- 显示本次会话持续时间。
- 显示当日累计会话时间。
- 能获得服务端用量时，显示实际 Token 或调用量。
- 暂时无法获得实时账单时，显示“估算费用”，不得标记为实际费用。
- 支持每日费用上限和余额不足提示。
- 达到用户设置的上限时停止新建会话。
- 服务端必须维护用户级、设备级、租户级和全局级四层额度。
- 费用判断不得只依赖火山引擎小时级账单，应基于会话事件实时生成内部用量账本。
- 应支持免费试用、付费套餐、按量计费、封顶和欠费停用策略。
- 管理员应能配置全局日预算和紧急熔断阈值。

### 7.9 错误处理

- 网络断开：停止发送，保留有限缓冲，超时后结束会话。
- 鉴权失败：停止重试并提示检查凭证。
- 欠费或配额不足：提示家属处理，不无限重连。
- 服务限流：按指数退避重试。
- 音频设备断开：立即暂停或结束会话。
- 麦克风权限被拒绝：展示授权入口。
- 播放失败：停止会话并保存诊断信息。
- 全局并发已满：进入排队、降级或提示稍后重试，不允许无上限冲击供应商。
- 单用户异常高频建连：限流并进入风险检查。
- 供应商区域性故障：停止扩大重试流量，执行熔断和退避。

### 7.10 规模化运营

- 支持 10 万至 100 万注册用户。
- 支持通过配置调整灰度人群、模型、音色、静默超时和功能开关。
- 支持按应用版本、手机型号、地区和用户分群灰度发布。
- 支持查看注册用户、日活、会话数、会话时长、峰值并发、成功率、延迟和成本。
- 支持用户投诉定位，能够通过脱敏会话 ID 查询链路日志。
- 支持冻结用户、吊销设备、撤销凭证和终止异常会话。
- 支持在供应商配额不足或费用异常时一键禁止新会话，但不影响用户查看本地状态。

### 7.11 V1/V2 模式管理

- 应用必须在一个主 Activity 内完成模式选择和内容切换。
- 模式选择控件始终可见，不要求进入二级设置页面。
- V1 内容区域使用封板版本的原有布局、文案和操作。
- V2 内容区域独立实现，不复用会引起 V1 行为变化的状态控件。
- 默认模式为 V2。
- 已安装旧版升级到支持模式切换的版本时，首次启动默认进入 V2；如需要降低迁移风险，可在发布前通过远程配置调整一次性迁移策略。
- 用户主动选择的模式优先于默认值，并持久化保存。
- 切换模式必须是原子操作：先停止旧模式，再切换 UI，最后启动新模式。
- 切换失败时保持旧模式停止状态并给出错误，不得让两套服务同时运行。
- V1 不使用 V2 的 API 凭证和付费额度。
- V2 不启动豆包 App，不执行无障碍点击。

## 8. 适老化要求

- 核心状态文字简短、高对比度。
- 一个屏幕内显示连接状态、会话按钮和必要配置。
- 老年模式下隐藏 API、模型和调试参数。
- 关键错误使用通俗中文。
- 日常使用不要求进入 Android 系统设置。
- 不以颜色作为唯一状态表达方式。
- 触控目标尺寸不低于 Android 推荐的可操作尺寸。

## 9. 隐私、安全与合规

- 首次采集麦克风前必须获得 Android 运行时授权。
- 界面必须持续显示当前是否正在采集和上传语音。
- 默认不保存原始录音。
- 如需保存录音或文本记录，必须单独获得用户同意。
- 永久密钥不得写入源码、APK、日志或崩溃报告。
- 服务端日志应脱敏，不记录完整 Token。
- 应向用户说明语音会发送至火山引擎处理。
- 需要遵守火山引擎服务协议、隐私规范及 Android 权限规范。

## 10. 非功能需求

### 10.1 性能

- 本地采集与播放附加延迟目标：不超过 200ms。
- 模型首包语音的目标体验延迟：不超过 2 秒，最终以服务能力和网络测试为准。
- 播放缓冲不得导致明显断续。
- 用户打断到停止模型播放的目标时间：不超过 500ms。

### 10.2 稳定性

- 连续运行 2 小时无崩溃。
- 连续完成 30 轮问答无不可恢复异常。
- 完成 20 次 Type-C 插拔恢复测试。
- 完成 20 次蓝牙断开与重连测试。
- 网络切换后能自动恢复或给出明确操作。

### 10.3 兼容性

- 首批基线：指定 vivo Android 16 手机。
- Type-C 基线：Newline NP-S2201。
- 蓝牙基线：当前项目已验证的双向通话设备。
- 后续扩展其他 Android 厂商和会议终端。

### 10.4 容量与可用性

- 注册用户设计容量：100 万。
- 日活、峰值并发和人均使用时长必须通过运营假设计算，不得直接用注册用户数代替并发数。
- 业务控制面月可用性目标不低于 99.9%；正式 SLA 需结合供应商 SLA、预算和部署方案确认。
- 单可用区故障不得造成全量用户不可用。
- 单个应用实例、数据库节点或缓存节点故障不得中断全部新会话。
- 服务接近供应商并发上限时，应在 70%、80%、90% 三档告警。
- 任何自动重试策略必须有上限、退避和随机抖动，避免百万客户端同时重试。

### 10.5 大规模性能

- 会话准入接口 P95 目标不超过 300ms，不包含供应商语音建连时间。
- 控制面所有接口必须支持幂等请求。
- 配置服务应允许客户端使用带版本号的本地缓存，控制面短时故障时不产生请求风暴。
- 推送和配置变更应分批下发，不得同时唤醒全部客户端。

## 11. 验收标准

### 11.1 独立性

- 卸载豆包 App 后，V2 仍能完整对话。
- 项目中不再使用豆包包名、豆包启动 Intent 或豆包无障碍点击。

### 11.2 基本对话

- 连接 Type-C 后能够启动实时会话。
- 用户语音被模型正确理解。
- 回复从 Newline 扬声器播放。
- 完成 30 轮连续问答。
- 用户可在回答中途打断。

### 11.3 蓝牙

- Type-C 未连接时，可选择并使用蓝牙通话设备。
- 蓝牙关闭时，系统提示开启，确认后继续选择。
- 已选择设备在列表首位显示。

### 11.4 费用控制

- 未开始会话时不上传音频。
- 结束或断开后停止云端用量。
- 本次和当日时长统计准确。
- 达到配置的费用或时长限制后不再自动建连。

### 11.5 安全

- APK 中搜索不到正式永久密钥。
- 日志中不出现完整 Token。
- 未授权麦克风时无法开始会话。

### 11.6 规模化验收

- 使用容量模型输出 10 万和 100 万用户下的平均并发、峰值并发和每日费用预测。
- 在供应商确认并发配额后完成 1.2 倍目标峰值的压测；语音媒体链路无法全量压测时，使用协议模拟器并抽样真音频验证。
- 证明单个用户或设备无法消耗整个租户额度。
- 证明同一会话请求重放不会重复扣减权益或创建多个云端会话。
- 验证 1%、5%、20%、50%、100% 灰度和快速回滚。
- 验证供应商限流、控制面故障和数据库故障时不会形成重试风暴。
- 验证全局费用熔断能够在内部用量达到阈值后阻止新会话。

### 11.7 模式切换验收

- 全新安装后首次打开默认显示 V2 界面。
- 选择 V1 后完整显示 `b92d16c` 封板版本对应的 V1 界面和能力。
- 选择 V2 后完整显示 V2 界面。
- 杀死应用、重新打开和重启手机后仍保持用户最后一次选择。
- V1 活动会话中切换到 V2，V1 会话先结束且不再执行无障碍点击。
- V2 活动会话中切换到 V1，V2 云端连接、采集和播放全部停止。
- 连续切换 20 次不存在两个前台服务、两个录音实例或两个云端会话并存。
- V1 模式测试不消耗 V2 API 用量。
- V2 模式在未安装豆包 App 时仍能正常运行。

## 12. 版本规划

### V2.0-P0：技术验证

- 目标用户：开发者和内部测试人员，1～5 人。
- 开通服务并运行官方实时语音 Demo。
- 确认 Android SDK 或 WebSocket API 能力。
- 验证音频格式、打断、单连接时长、并发和计费。
- 验证 Newline 和蓝牙路由。
- 允许使用受限、可撤销的测试凭证，但不得提交 Git。
- 不建设正式账号、套餐、运营后台和分布式基础设施。
- P0 技术验证可以使用独立开发入口；完成验证后再接入统一模式选择界面。

退出条件：

- 不依赖豆包 App 完成端到端实时语音。
- Type-C 和蓝牙各完成至少 10 轮对话。
- 明确 SDK/WebSocket 选型、单连接限制和基本成本。

### V2.0-P1：最小可用版本

- 目标用户：10～100 名受控内测用户。
- 在本 App 内完成实时双向语音。
- 提供开始、结束、状态和错误提示。
- 移除豆包 App 自动化依赖。
- 建设一个最小服务端，负责保存正式密钥和签发会话配置。
- 增加基础用户标识、设备标识、用量记录和每日限额。
- 使用单体服务和托管数据库即可，不提前拆分微服务。
- 在一个主界面中加入 V1/V2 模式选择，默认 V2。
- 将当前 `b92d16c` 版本作为 V1 封板基线，建立回归测试，防止 V2 开发改变 V1 行为。

退出条件：

- 100 名用户范围内稳定运行。
- 无正式永久密钥进入 APK。
- 能按用户限制时长或费用。
- 具备基础错误、延迟和用量监控。

### V2.0-P2：费用与稳定性

- 目标用户：100～1 万注册用户。
- 增加用量、费用上限、静默暂停和自动重连。
- 完成长时间、插拔和网络异常测试。
- 增加账号、套餐、支付或权益、供应商账单对账。
- 增加远程配置、灰度发布和客服诊断。
- 根据实际负载决定是否增加缓存和消息队列。

退出条件：

- 峰值负载和费用预测可用。
- 灰度、回滚、限流和费用熔断通过验证。
- 单体服务的容量和拆分阈值已经量化。

### V2.0-P3：规模化平台

- 目标用户：1 万～10 万注册用户。
- 建设用户、设备、权益、会话准入和内部用量账本。
- 建设多可用区部署、限流、熔断、灰度和运营后台。
- 与火山引擎确认目标并发、扩容周期、商务价格和 SLA。
- 完成容量测试、安全测试和成本压力测试。

进入条件：

- 实际峰值并发接近现有架构安全容量的 50%。
- 单体数据库、缓存或服务已出现可量化瓶颈。
- 商业收入和增长预测能够支持规模化基础设施成本。

### V2.0-P4：百万用户扩展

- 目标用户：10 万～100 万注册用户。
- 按真实流量模型采购供应商并发。
- 完善多可用区、必要时多地域部署。
- 完善实时用量账本、反滥用、自动扩缩和灾难恢复。
- 完成百万用户容量模型、故障演练和成本压力测试。

### V2.1：适老化增强

- 本地唤醒词。
- 更完善的家属模式。
- 对话摘要、提醒和可选记录。

## 12.1 平滑迁移约束

为保证从 P0 平滑迁移到 P4，以下接口从 P0 开始固定边界：

- `RealtimeVoiceClient`：隔离官方 SDK 和 WebSocket。
- `CredentialProvider`：隔离本地测试凭证和服务端短期凭证。
- `SessionCoordinator`：统一会话状态与重连。
- `AudioRouteController`：统一 Type-C 和蓝牙路由。
- `UsageTracker`：统一本地估算和服务端正式账本。
- `AppModeCoordinator`：统一 V1/V2 模式持久化、停止旧模式和启动新模式。

阶段升级允许替换内部实现，但 UI 和主要业务流程不直接依赖供应商协议。

## 13. 外部依赖与待确认事项

- 火山引擎账号已完成实名认证。
- 豆包端到端实时语音服务已开通。
- 已获得测试 AppID、Access Token 或服务要求的其他凭证。
- 已确认可用模型版本、音色 ID、Resource ID 和并发配额。
- 需要通过官方文档或商务确认单连接最长时长、重连策略及正式价格。
- 需要验证 Android SDK 是否支持外部 PCM 输入输出；如果 SDK 只能内部采播，需要验证其是否稳定遵循 `setCommunicationDevice`。
- 需要向火山引擎提供 10 万和 100 万用户的容量预测，确认可申请的并发/QPS上限、扩容提前量和峰值保障方式。
- 需要确认供应商是否提供适合移动端的短期凭证；没有安全短期凭证时必须设计媒体网关或供应商认可的客户端授权方案。
- 需要获得供应商正式 SLA、故障通知和技术支持渠道。
- 需要确定商业模式，否则无法设置用户级权益、每日预算和全局成本上限。
