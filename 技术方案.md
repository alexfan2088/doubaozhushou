# Android 原生服务技术方案

> 本文档是 V1 豆包 App 兼容模式的历史技术基线。当前 V2 实现请以
> `技术方案V2.md` 和 `P0验证记录.md` 为准。

## 1. 方案概述

在 Android 16 手机上安装一个原生应用，通过前台服务监听 Newline NP-S2201 的 USB Type-C 连接。当目标设备接入后，服务检测 USB 音频输入、输出，申请音频焦点，启动豆包并直接进入连续对话。

本应用是连接协调器，不代理或转发豆包的原始音频。优先让 Android 将 Newline 注册为标准 USB 音频设备，由豆包直接录制和播放。

已有朋友开发的 Android 程序实现了“打开程序后自动拉起豆包，通过 Newline 连续对话”。该程序作为本项目的行为基线，首要任务是记录并复现其启动协议、权限配置和音频路由行为。

## 2. 建议技术栈

- Kotlin
- Android SDK，最低版本根据目标设备确定，目标版本为 Android 16
- Foreground Service
- BroadcastReceiver 或动态注册的 USB 事件监听
- UsbManager
- AudioManager、AudioDeviceCallback
- Jetpack Compose，用于首次配置和家属管理界面
- DataStore，用于保存目标设备标识和配置状态

## 3. 核心模块

### UsbConnectionMonitor

监听 USB 连接和断开，识别 Vendor ID、Product ID 与接口描述，过滤非目标设备。

### AudioDeviceMonitor

通过 `AudioManager` 和 `AudioDeviceCallback` 监听 USB 输入、输出设备是否完成注册，记录当前路由状态。

### SessionCoordinator

实现连接状态机：

`空闲 → 检测设备 → 等待授权 → 等待音频设备 → 启动豆包 → 对话中 → 断开并恢复`

统一处理超时、重试、断线和状态提示，避免多个广播重复启动会话。

### DoubaoLauncher

封装参考程序已验证的豆包启动方式，包括包名、入口 Activity、Intent Action、URI、Extras 和必要标志位，并按豆包版本维护兼容配置。

### ElderlyStatusNotifier

提供常驻通知、大字状态页和简短语音提示。详细错误写入家属诊断页面，不直接暴露给老年用户。

## 4. 必须先验证的技术问题

1. NP-S2201 的 Type-C 端口是否枚举出 USB 音频输入和输出。
2. 手机与 NP-S2201 的 USB Host/Device 角色能否正确协商。
3. vivo Android 16 是否默认将媒体输出和通信录音切换到该 USB 设备。
4. 豆包是否跟随系统 USB 音频路由。
5. 参考程序通过哪种 Intent、URI、Activity 或其他机制直接进入连续对话。
6. USB 插入事件能否在目标 vivo 系统上可靠触发前台服务及应用启动。

## 5. 参考程序分析输入

优先级从高到低：

1. 经作者授权提供的完整源码和构建文件。
2. APK、正常运行的测试手机及作者说明。
3. APK 包信息、Manifest、`adb logcat`、`dumpsys activity`、`dumpsys audio` 和 USB 枚举记录。
4. 启动前后屏幕录像及完整操作步骤。

拿到参考程序后，应先建立可重复的基线测试，不应仅凭界面现象猜测实现方式。

## 6. 实现原则

- 不使用隐藏 Android API 作为量产依赖。
- 不假设普通服务可以强制改变豆包的录音设备。
- 以参考程序的已验证行为为基准，同时将版本依赖集中隔离，便于豆包升级后适配。
- 首次权限由家属完成，日常流程尽量做到插线即用。
- 每个关键步骤都应有超时、重试、日志和可理解的用户提示。
- 首先制作真机技术验证版本，再扩展完整界面和适老化功能。
