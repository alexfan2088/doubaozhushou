# 豆包助手 V2.0 技术方案

> 文档版本：V2.0 Draft 4（V1/V2 单界面模式切换），2026-06-30

## 0. 当前实现快照（2026-07-02）

- 媒体链路采用火山引擎官方 Android SDK `0.0.14.6.1-bugfix`，使用 SDK 内置录音、
  播放和 AEC，模型版本固定为 `1.2.1.1`。
- `V2Activity` 当前负责界面和会话编排，`V2VoiceForegroundService` 负责麦克风前台
  服务状态；目标架构中的 Service 持有完整会话尚未完成。
- 路由使用 `AudioRouteManager`，优先级为 Type-C、已选蓝牙、无设备；当前产品行为
  允许外设无拾音能力时回退手机麦克风。
- 本地唤醒使用 Sherpa-ONNX；云端无输出达到配置时间后断开，再由“豆包豆包”唤醒。
- Token 节省设置使用独立 SharedPreferences 持久化：本地 TTS WAV 欢迎词、回答句数、
  静默时间、唤醒冷却、上下文轮数及回答时限。
- 用量使用 SDK Usage 事件在本机累计并估算费用，尚不等同于供应商正式账单。
- 测试凭证仍由本机构建配置注入 APK，只适用于 P0 受控测试。

## 1. 方案概述

V2 在 Android App 内直接接入火山引擎豆包端到端实时语音服务，负责音频采集、实时传输、模型事件处理、音频播放、设备路由、会话生命周期和费用控制。

V2 不启动豆包 App，不使用无障碍服务点击豆包控件，也不依赖豆包 App 的每日实时语音额度。

本方案按 10 万至 100 万注册用户设计。注册用户数不等于实时并发数，正式容量必须根据日活、人均使用时长、峰谷分布和增长目标计算，并提前向火山引擎采购或申请足够的并发/QPS。

长期容量目标不代表当前立即建设百万用户基础设施。当前先完成 P0 单机验证和 P1 小规模内测，只保留必要的模块边界；缓存、消息队列、微服务、多地域和复杂账务系统均由真实指标触发。

## 1.1 渐进式架构原则

- P0 优先跑通官方能力，不建设业务后台。
- P1 使用模块化单体服务，不使用微服务。
- P2 根据真实用户和费用数据补齐商业能力。
- P3 在出现明确容量瓶颈前后拆分关键服务。
- P4 才建设百万用户所需的完整平台能力。
- 客户端模块接口保持稳定，服务端实现可以逐步替换。
- 小规模阶段也不把永久生产密钥写入正式 APK。

## 1.2 V1 封板与双模式原则

- `b92d16c` 作为 V1 封板基线。
- V1 源码继续保留在同一 App 内，但不再承接 V2 新需求。
- V2 使用独立的 UI 状态、服务协调器和语音实现。
- 一个主 Activity 展示模式选择及对应内容。
- 默认模式为 V2，用户选择持久化。
- 任意时刻只允许一个模式处于运行状态。
- 模式切换必须先完成旧模式资源释放，再启动新模式。
- V1 回归测试和 V2 测试分开执行。

整体链路：

```text
Newline/蓝牙麦克风
        │ PCM
        ▼
Android AudioRecord 或官方 SDK 采集层
        │
        ▼
ConversationSession
        │ WebSocket / 官方 Android SDK
        ▼
豆包端到端实时语音大模型
        │ 流式音频与事件
        ▼
Android AudioTrack 或官方 SDK 播放层
        │
        ▼
Newline/蓝牙扬声器
```

## 2. 技术选型

### 2.1 推荐方案

优先使用官方“端到端 Android SDK”，前提是技术验证确认：

- 支持目标 Android 版本和 ABI。
- 支持端到端实时语音模型。
- 支持外部 PCM 输入输出，或者内部采播能够遵循本应用选择的 Type-C/蓝牙通信设备。
- 暴露会话、用户说话、模型回答、打断、错误和用量相关事件。
- 可配置模型、音色、人设和判停参数。

官方文档已经列出端到端实时语音 API、Android SDK 和 iOS SDK入口，但具体依赖版本、授权参数和接口行为必须以开通账号后可见的最新文档和 Demo 为准。

### 2.2 备选方案

如果官方 Android SDK 无法稳定控制外接音频设备，则使用 WebSocket API，自行实现：

- `AudioRecord` 采集。
- 音频重采样和分帧。
- WebSocket 二进制协议。
- 会话事件编解码。
- `AudioTrack` 流式播放。
- 打断、缓冲和重连。

WebSocket 方案控制力更强，但协议、音频时序和稳定性开发成本更高。

### 2.3 不推荐方案

不建议首版使用“流式 ASR + 文本大模型 + 流式 TTS”的三级级联方案。该方案虽然组件透明，但需要自行处理：

- 判停。
- 对话上下文。
- 首包延迟。
- 打断。
- ASR 与 TTS 并行取消。
- 情绪和副语言信息丢失。

只有在端到端模型无法满足业务能力、成本或可控性要求时，再将级联方案作为替代路线。

## 3. 系统架构

### 3.1 Android 客户端

```text
UI
 ├─ ConnectionStatus
 ├─ ConversationControls
 ├─ VoiceProfileSettings
 └─ UsageAndDiagnostics

ForegroundService
 └─ SessionCoordinator
     ├─ AudioRouteController
     ├─ RealtimeVoiceClient
     ├─ AudioCaptureEngine
     ├─ AudioPlaybackEngine
     ├─ ConversationStateMachine
     ├─ UsageTracker
     └─ RecoveryPolicy

Local persistence
 ├─ DataStore: 用户配置
 ├─ Room: 会话与用量统计（可选）
 └─ Encrypted storage: 短期凭证
```

### 3.2 服务端

P1 正式内测建议增加轻量服务端：

```text
Android App
    │ HTTPS，用户/设备鉴权
    ▼
业务服务端
    ├─ 凭证与权限控制
    ├─ 临时会话授权
    ├─ 用户额度与每日预算
    ├─ 配置下发
    └─ 用量审计
        │
        ▼
火山引擎豆包语音服务
```

加入模式层后的客户端结构：

```text
MainActivity
 ├─ ModeSelector
 │   ├─ V1 豆包 App 模式（兼容版）
 │   └─ V2 豆包 API 模式（默认/推荐）
 └─ ModeContent
     ├─ V1Screen
     │   └─ V1BridgeController
     │       ├─ 现有 AudioRouteManager
     │       ├─ 现有 DoubaoLauncher
     │       └─ 现有 DoubaoAccessibilityService
     └─ V2Screen
         └─ V2SessionCoordinator
             ├─ AudioRouteController
             ├─ RealtimeVoiceClient
             ├─ CredentialProvider
             └─ UsageTracker

AppModeCoordinator
 ├─ 保存当前模式
 ├─ stop(V1/V2)
 ├─ 等待资源释放
 └─ start(V1/V2)
```

V1 与 V2 可以共享纯设备发现代码，但不能直接共享带生命周期和副作用的会话状态。

服务端部署方式取决于官方鉴权能力：

1. 如果官方支持安全的短期客户端凭证：服务端只签发短期凭证，Android 直连语音服务。
2. 如果只能使用长期 Access Token：生产环境由服务端代理 WebSocket，或采用官方认可的移动端授权方案。
3. 原型阶段可以在本地测试配置中使用测试 Token，但禁止提交 Git 和发布正式 APK。

当规模达到 10 万至 100 万用户时，“轻量服务端”必须升级为独立控制面。默认推荐客户端直连供应商媒体面，业务服务端只负责身份、授权、准入、配置和用量：

```text
                         ┌────────────────────────────┐
Android App ──HTTPS────► │ 自有控制面                  │
                         │ API Gateway / WAF           │
                         │ Identity & Device           │
                         │ Entitlement & Admission     │
                         │ Credential Broker           │
                         │ Config & Feature Flags      │
                         │ Usage Ledger & Billing      │
                         └─────────────┬──────────────┘
                                       │ 配额/监控/对账
                                       ▼
                              火山引擎管理接口

Android App ═════ 实时音频直连 ═════► 豆包实时语音媒体面
```

如果供应商不提供安全的移动端短期凭证，不能把租户级长期 Token 下发到百万台设备。此时有两个选择：

1. 与供应商协商正式的移动端授权或设备级凭证方案，这是首选。
2. 建设媒体代理网关，由自有服务端持有供应商凭证；该方案成本和运维复杂度显著更高。

不建议默认代理全部音频。以单声道 16kHz、16bit 原始 PCM 粗略估算，单方向约 256kbps，双向 1 万并发约 5.12Gbps，尚未计算协议、TLS、副本和跨可用区流量。实际编码可能降低带宽，但媒体代理仍需专项容量设计。

### 3.3 各阶段服务端形态

| 阶段 | 用户规模 | 服务端形态 | 暂不建设 |
|---|---:|---|---|
| P0 | 1～5 | 无正式服务端；本地受限测试配置 | 账号、计费、微服务 |
| P1 | 10～100 | 单体 API + 托管数据库 | 消息队列、多地域、复杂运营后台 |
| P2 | 100～1 万 | 模块化单体 + 缓存；按需增加异步任务 | 全面微服务化 |
| P3 | 1 万～10 万 | 多可用区控制面，拆分准入、权益和用量 | 无业务依据的跨地域部署 |
| P4 | 10 万～100 万 | 完整控制面、事件流、实时账本和灾备 | 无 |

技术升级必须由指标触发，而不是仅由注册用户数触发：

- API CPU、内存或连接数持续超过安全水位。
- 数据库 P95 延迟或写入吞吐接近容量上限。
- 用量事件影响主会话链路。
- 单次发布风险无法通过现有灰度控制。
- 单可用区故障影响达到不可接受程度。

## 4. 核心模块设计

### 4.0 `AppModeCoordinator`

职责：

- 读取和保存当前模式。
- 新安装或无配置时返回 V2。
- 接收用户模式切换请求。
- 阻止重复和并发切换。
- 调用旧模式 `stop()` 并等待资源释放完成。
- 确认麦克风、音频焦点、前台服务和网络会话均已释放。
- 更新界面内容。
- 初始化并启动新模式。
- 切换失败时进入安全停止状态。

建议接口：

```kotlin
enum class AppMode { V1_COMPAT, V2_API }

interface ModeRuntime {
    suspend fun start()
    suspend fun stop()
    val state: StateFlow<ModeRuntimeState>
}

interface AppModeCoordinator {
    val selectedMode: StateFlow<AppMode>
    suspend fun switchTo(mode: AppMode): Result<Unit>
}
```

切换顺序：

```text
用户选择新模式
  → 禁用模式选择控件
  → 停止旧模式自动触发
  → 结束旧会话
  → 停止录音和播放
  → 释放通信设备和音频焦点
  → 停止旧模式前台服务
  → 保存新模式
  → 切换界面
  → 初始化新模式
  → 恢复模式选择控件
```

禁止只切换 View 的显示状态而不停止旧模式后台逻辑。

### 4.1 `AudioRouteController`

职责：

- 监听 Type-C、蓝牙和系统音频设备变化。
- 维护 Type-C > 已选蓝牙 > 无设备的优先级。
- 设置 `AudioManager.MODE_IN_COMMUNICATION`。
- 使用 `setCommunicationDevice` 选择通信设备。
- 向采集和播放引擎提供有效的输入、输出设备。
- 设备失效时立即通知会话协调器。

关键规则：

- Type-C 必须同时具备输入和输出。
- 蓝牙必须具备通话能力，不能只有 A2DP。
- 不长期缓存已断开的 `AudioDeviceInfo`。
- 路由切换期间停止或暂停采播。

### 4.2 `AudioCaptureEngine`

WebSocket 自采集模式下使用 `AudioRecord`：

- 音源优先使用 `VOICE_COMMUNICATION`，并在真机验证 AEC 行为。
- 使用服务要求的采样率、PCM 位深和声道数。
- 固定时长分帧，保持单调递增时间戳或序列号。
- 音频线程禁止执行网络阻塞操作。
- 使用有界队列，拥塞时丢弃旧帧并记录指标，不能无限增长。
- 仅在 `LISTENING` 或服务要求的双工状态中采集。

SDK 内部采集模式下：

- 在启动 SDK 前完成通信设备选择。
- 通过 `dumpsys audio`、录音回环和物理遮挡测试确认实际输入设备。
- 监测路由变化并在必要时重启 SDK 音频引擎。

### 4.3 `RealtimeVoiceClient`

对官方 Android SDK 或 WebSocket 实现统一接口：

```kotlin
interface RealtimeVoiceClient {
    suspend fun connect(config: SessionConfig)
    suspend fun sendAudio(frame: AudioFrame)
    suspend fun interrupt()
    suspend fun finishInput()
    suspend fun disconnect(reason: DisconnectReason)
    val events: Flow<RealtimeVoiceEvent>
}
```

事件至少包括：

- 连接成功。
- 会话创建成功。
- 用户开始说话。
- 用户停止说话。
- 模型开始回答。
- 模型音频分片。
- 模型回答结束。
- 用户打断。
- 用量信息。
- 可重试错误。
- 不可重试错误。

### 4.4 `AudioPlaybackEngine`

WebSocket 自播放模式使用 `AudioTrack`：

- 流式模式播放模型音频。
- 使用有界 jitter buffer。
- 首包达到最低缓冲后开始播放。
- 打断时执行 flush，而不是等待剩余音频播放完。
- 每次会话带 generation ID，丢弃旧 generation 音频。
- 播放设备变化时重建 `AudioTrack`。

### 4.5 `SessionCoordinator`

负责所有模块编排，禁止 UI、广播接收器或音频回调直接操作完整会话。

状态机：

```text
IDLE
  └─ device ready → DEVICE_READY
DEVICE_READY
  └─ user/auto start → CONNECTING
CONNECTING
  ├─ success → LISTENING
  └─ failure → ERROR/RETRY_WAIT
LISTENING
  ├─ user speech → USER_SPEAKING
  ├─ model response → MODEL_SPEAKING
  ├─ pause → PAUSED
  └─ stop/device lost → STOPPING
USER_SPEAKING
  ├─ end speech → WAITING_RESPONSE
  └─ stop/device lost → STOPPING
WAITING_RESPONSE
  ├─ audio response → MODEL_SPEAKING
  └─ timeout → ERROR
MODEL_SPEAKING
  ├─ user interrupt → USER_SPEAKING
  ├─ response end → LISTENING
  └─ stop/device lost → STOPPING
PAUSED
  ├─ resume → CONNECTING/LISTENING
  └─ stop → STOPPING
STOPPING
  └─ cleanup complete → DEVICE_READY/IDLE
```

约束：

- 同一时间最多一个活动会话。
- 用户主动结束后禁止自动重启。
- 设备断开优先级高于重连。
- 鉴权、欠费和参数错误不得无限重试。

### 4.6 `UsageTracker`

记录：

- 会话开始和结束时间。
- 实际连接时长。
- 用户音频上传时长。
- 模型音频下行时长。
- 服务端返回的输入、输出 Token 或计量值。
- 重连次数和错误原因。

本地估算与正式账单分开显示：

- `estimatedCost`：根据本地统计和当前单价估算。
- `billedCost`：仅使用服务端账单或用量 API 数据。

## 5. 音频参数与双工策略

### 5.1 参数发现

在 P0 阶段从官方 Demo 和接入文档确认：

- 输入采样率。
- 输出采样率。
- PCM/Opus 等编码。
- 单双声道。
- 单帧建议时长。
- 是否要求连续发送静音帧。
- 是否支持客户端 VAD。
- 是否支持服务端 VAD 和判停参数。

不得在参数未确认前将 16kHz PCM 固化为正式协议。

### 5.2 回声消除

优先级：

1. Newline 硬件 AEC。
2. 官方 SDK 内置 AEC。
3. Android `AcousticEchoCanceler`。
4. 应用层 AEC。

同一链路避免同时启用多个强 AEC，防止双重处理导致声音失真。需要分别测试 Type-C 和蓝牙路径。

### 5.3 打断

模型回答期间保持可打断采集：

1. 检测用户语音或接收服务端用户说话事件。
2. 向模型发送 interrupt/cancel。
3. 清空本地待播放音频。
4. 将状态切换至 `USER_SPEAKING`。
5. 保留已经确认的对话上下文。

## 6. 会话时长与费用控制

API 付费消除了豆包 App 的每日 60 分钟产品额度，但不意味着可以无成本全天保持云端流。

建议策略：

- 默认由用户点击“开始对话”建立云端会话。
- 用户点击结束或设备断开时立即断开。
- 当前默认静默超时为 15 秒，可配置为 10、15、30 或 60 秒。
- 静默超时后进入 `PAUSED`，停止上传和计费连接。
- V2.1 使用本地唤醒词恢复，避免云端全天监听。
- 设置单次最长会话和每日预算。
- 在官方单连接到期前主动建立新连接并迁移必要上下文。
- 当前可配置保留上下文，或在 5、8、10、15、20 轮后无欢迎词重建会话；默认 10 轮。
- 当前欢迎词由 APK 内置 Sherpa-ONNX 和中文 VITS 模型生成 WAV，支持 5 个音色；
  模型随 APK 安装，不需要运行时下载。对 16-bit PCM 执行增益和限幅后按文本及
  音色缓存；本地播放完成后才启动 SDK 录音，避免欢迎词被回采。该流程不修改系统
  媒体音量，离线引擎异常时回退 Android 系统 TTS。

需在 P0 阶段实测以下计费边界：

- 静音是否产生输入语音 Token。
- WebSocket 连接但不发音频是否计费。
- 模型回答音频如何计量。
- 单连接最长时长。
- 断线重连是否创建新计费会话。

### 6.1 分层准入

每次建立供应商会话前必须调用会话准入服务。准入顺序：

1. 校验用户身份和设备绑定。
2. 校验应用版本和风险状态。
3. 校验用户套餐、余额和每日上限。
4. 校验用户和设备并发。
5. 校验租户全局并发。
6. 校验供应商实时配额余量。
7. 原子扣减或预占一个并发席位。
8. 签发短期会话授权。

会话结束、超时或心跳丢失后释放席位。所有操作必须使用会话 ID 幂等，避免客户端重试重复占用。

### 6.2 实时用量账本

供应商后付费账单可能按小时生成并存在延迟，不能作为实时成本熔断依据。自有平台需要：

- 在会话创建时记录预占。
- 按固定周期或供应商事件累计连接时长和 Token。
- 会话结束时结算内部用量。
- 定期与供应商 UsageMonitoring 或账单对账。
- 将估算、已确认和差异分开保存。
- 发现差异超过阈值时暂停高风险流量。

## 7. 鉴权与密钥安全

### 7.1 原型

- 凭证写入未跟踪的 `local.properties` 或本机环境变量。
- 构建时注入仅供测试的低权限凭证。
- 限制服务额度、来源和有效期。
- 不提交 Git。
- P0 凭证只允许少量测试设备使用，并设置低额度和随时可撤销。

### 7.2 正式版

- App 先登录业务服务端。
- 服务端验证用户和设备。
- 服务端签发短期会话授权或代理语音连接。
- 长期 Access Token 只保存在服务端密钥系统。
- Android 仅缓存短期凭证，并使用 Android Keystore 保护。
- 日志统一脱敏，仅保留 Token 前后少量定位信息或哈希。

P1 的服务端无需微服务化。一个结构清晰的单体应用即可包含：

- 用户和设备校验。
- 会话配置签发。
- 每日时长或费用上限。
- 基础用量写入。
- 管理员停用开关。

当供应商无法提供安全的客户端短期凭证时，P1 必须先与供应商确认移动端授权方式。不能因为测试用户较少，就将长期生产 Token 发布到 APK。

### 7.3 大规模反滥用

- 使用用户登录态和设备密钥双重绑定。
- 每次授权包含用户 ID、设备 ID、会话 ID、过期时间、随机数和允许的模型范围。
- 服务端拒绝重放、过期和跨设备授权。
- 接入 Android 应用完整性校验作为风险信号，但不能将其作为唯一身份依据。
- 按用户、设备、IP、ASN、版本和地区设置分层限流。
- 对异常长会话、高频重连、多设备共享账号和费用突增进行风控。
- 支持秒级吊销用户、设备和授权。

## 8. Android 生命周期

- 对话期间使用合规的麦克风类型前台服务。
- Manifest 声明 `RECORD_AUDIO`、网络和相应前台服务权限。
- 开始采集前检查运行时麦克风权限。
- 屏幕关闭后是否继续对话由明确设置控制。
- 服务被系统终止后，不自动恢复用户已主动结束的会话。
- 重启应用时恢复配置，但不默认恢复收费会话。
- 电话抢占音频焦点时暂停采播，通话结束后提示或自动恢复。
- 模式偏好独立于会话状态持久化。
- 恢复应用时只创建已选择模式的运行时。
- 默认选择 V2 不等于自动建立收费会话；是否自动开始由 V2 的独立设置决定。
- V1 的广播监听和无障碍自动请求只允许在 V1 模式启用。
- V2 模式下必须取消 V1 所有待执行的 Handler、广播触发和无障碍点击请求。

## 10. 错误分类与恢复

| 错误类型 | 示例 | 策略 |
|---|---|---|
| 瞬时网络错误 | Wi-Fi 切换、短时超时 | 指数退避，最多有限次数 |
| 设备错误 | Type-C/蓝牙断开 | 立即停止采播，不重连云端 |
| 鉴权错误 | Token 无效、过期 | 刷新一次，失败后停止 |
| 配额错误 | 欠费、额度不足 | 停止并提示家属 |
| 限流 | 并发或 QPS 超限 | 按服务端建议延迟重试 |
| 协议错误 | 不支持的参数或事件 | 停止，记录请求 ID |
| 音频错误 | 初始化失败、设备失效 | 重建一次，失败后结束 |

所有服务端错误应记录官方请求 ID 或 log ID，便于提交工单。

## 15. 容量模型

### 15.1 计算方法

注册用户不能直接转换为并发。至少需要以下输入：

- 注册用户数 `R`。
- 日活率 `D`。
- 每个日活用户每日平均实时语音分钟数 `M`。
- 峰值系数 `P`。
- 并发安全系数 `S`。

平均并发估算：

```text
average_ccu = R × D × M ÷ 1440
```

规划峰值并发：

```text
planned_peak_ccu = average_ccu × P × S
```

示例仅用于说明，不能代替真实运营数据。假设日活率 20%、每个日活用户每天使用 20 分钟、峰值系数 4、安全系数 1.3：

| 注册用户 | 日活 | 平均并发 | 规划峰值并发 |
|---:|---:|---:|---:|
| 10 万 | 2 万 | 约 278 | 约 1,445 |
| 100 万 | 20 万 | 约 2,778 | 约 14,445 |

还必须额外评估极端活动场景。如果 10% 注册用户在同一时间上线，10 万用户对应 1 万并发，100 万用户对应 10 万并发，远高于普通日均模型。

### 15.2 供应商容量

- 通过官方配额接口持续读取 `value` 和 `limit`。
- 不能假设默认开通配额能够承载商业规模。
- 上线前向火山引擎提交用户增长、峰值并发、地域和时长预测。
- 明确扩容申请提前量、突发容量、限流错误码和商务保障。
- 生产与测试应用、配额和凭证分离。
- 如需拆分多个 AppID，必须由供应商确认其合规性，并在内部保持统一用户和账单归属。

### 15.3 控制面容量

控制面流量远低于媒体流量，但存在连接风暴：

- 客户端启动和会话建连必须加入随机抖动。
- 配置使用 CDN/边缘缓存和 ETag。
- 准入服务保持无状态并水平扩展。
- 并发席位使用支持原子操作的集中式存储，并设计存储故障降级策略。
- 用量事件通过消息队列异步写入账本，避免主链路同步写热点数据库。
- 数据库按用户或会话维度分区，避免单表无限增长。

## 16. 高可用与灾难恢复

- 控制面至少跨三个可用区部署。
- API 网关、准入和凭证服务无状态化。
- Redis 等并发状态组件使用高可用部署，并避免将其作为唯一账本。
- 权益与费用账本以持久数据库为准，事件总线支持重放。
- 远程配置支持版本、签名、回滚和本地最后可用缓存。
- 供应商故障时默认停止新会话，避免客户端无限重试。
- 是否跨地域部署取决于用户分布、供应商接入点、数据合规和成本。
- 制定并演练 RTO、RPO；正式指标由业务等级确定。

## 17. 发布策略

- 新模型、新 SDK、新音色和协议参数全部使用远程开关。
- 按内部用户、1%、5%、20%、50%、100% 分阶段发布。
- 每阶段观察成功率、延迟、崩溃、费用和投诉。
- 支持按版本、机型、地区和设备类型回滚。
- 禁止未经灰度同时唤醒全部客户端或切换全部模型。
- 客户端必须保留最低支持版本和强制升级策略。

## 18. 风险

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| SDK 不支持指定外接输入输出 | 无法稳定使用 Newline | P0 同时验证外部 PCM WebSocket 方案 |
| 单连接存在时长限制 | 长会话中断 | 到期前主动重连并恢复上下文 |
| 静音仍计费 | 成本不可控 | 本地静默超时和唤醒策略 |
| Token 暴露 | 费用和数据风险 | 服务端短期凭证或代理 |
| 蓝牙 SCO 音质较低 | 识别和体验下降 | 优先 Type-C，实测 WBS/BLE Audio |
| 回声导致自激对话 | 错误打断和费用增加 | 验证硬件 AEC，必要时半双工降级 |
| 厂商后台限制 | 锁屏后会话停止 | 前台服务、白名单指引和真机测试 |
| 官方 API 或价格变化 | 功能或成本变化 | 配置化模型和价格，不硬编码 |
| 默认并发不足 | 高峰大量用户无法建连 | 提前采购容量、分层准入和排队 |
| 百万客户端同时重试 | 雪崩并扩大供应商故障 | 指数退避、随机抖动、全局熔断 |
| 共享 Token 泄露 | 全租户费用和数据风险 | 短期授权、设备绑定、服务端吊销 |
| 账单延迟 | 成本超限后才发现 | 内部实时用量账本和全局预算 |
| 媒体全量代理成本过高 | 带宽和运维不可控 | 优先客户端直连供应商媒体面 |
| 单区域故障 | 大面积不可用 | 多可用区，评估多地域和供应商保障 |
| 两种模式同时运行 | 双录音、重复播放和重复计费 | 单一 `AppModeCoordinator`、互斥锁和资源释放确认 |
| V2 开发破坏 V1 | 兼容用户无法使用 | V1 封板基线、独立控制器和回归测试 |
| 升级后用户困惑 | 不清楚两种模式差异 | V2 默认/推荐标识，V1 标记兼容模式 |

## 19. P0 开始前所需资料

- 火山引擎测试账号或已创建的语音应用。
- AppID。
- 测试 Access Token 或官方要求的鉴权材料。
- Resource ID。
- 模型版本选择：S2S-Omni 或 S2S-Strong Character。
- 音色 ID。
- 当前并发和配额。
- 官方 Android SDK Demo、依赖地址及版本。
- 目标 vivo 手机、Newline NP-S2201 和蓝牙设备。
- 10 万和 100 万用户下的 DAU、每日使用分钟数、峰值时段和增长预测。
- 火山引擎可提供的默认并发、目标并发、扩容周期、SLA 和商务报价。
- 是否允许客户端直连以及正式移动端鉴权方案。

不要通过聊天或 Git 提供主账号永久 AK/SK。测试凭证应使用最小权限、低额度和可随时撤销的配置。
