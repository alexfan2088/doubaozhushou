# Newline 会议终端与 Android 手机豆包连续对话需求说明

## 1. 需求背景

用户使用 USB Type-C 数据线连接 Android 16 手机（例如 vivo 手机）与 Newline NP-S2201 会议终端后，希望直接使用会议终端的麦克风和扬声器与手机端豆包进行连续语音对话，无需每轮手动切换音频设备或重新发起录音。

## 2. 产品定位

本项目开发一个安装在 Android 16 手机上的原生 Android 应用。应用的核心是一个可长期运行的前台服务，不重新实现豆包，而是负责识别 Newline 设备、配置音频链路、启动豆包并监控连接状态。

目标用户主要是老年人。用户回家后只需用 Type-C 线连接手机和 Newline NP-S2201，无需进入系统设置或反复点击按钮，即可在家中 Newline 设备拾音范围内与豆包连续对话。

## 3. 建设目标

连接成功后，系统自动将 Newline NP-S2201 识别为手机端的外接音频输入、输出设备：

- Newline NP-S2201 的麦克风负责采集用户语音。
- 采集到的语音传输至 Android 手机上的豆包应用。
- 豆包的语音回复通过 Newline NP-S2201 的扬声器播放。
- 支持用户与豆包多轮、连续语音对话。
- Type-C 连接或断开时自动完成音频路由切换，不要求用户每轮操作。
- 自动拉起豆包并进入可进行连续语音对话的状态。
- 日常使用遵循“一次安装配置，后续插线即用”的原则。

## 4. 使用场景

1. 家属首次安装应用，完成麦克风、通知、USB 设备访问、后台运行等授权，并配置豆包。
2. 老年用户回家后，使用支持数据和音频传输的 USB Type-C 线连接手机与 Newline NP-S2201。
3. 原生服务收到 USB 设备连接事件，校验设备身份和音频能力。
4. 服务将手机的语音输入和输出切换至 Newline NP-S2201。
5. 服务自动启动豆包，并在能力允许时直接进入连续语音对话。
6. 用户在 Newline 拾音范围内说话，会议终端将音频传给手机端豆包。
7. 豆包的语音回复从 Newline NP-S2201 扬声器播放。
8. 系统自动重复“拾音—回复—继续拾音”，用户不需要操作手机。
9. Type-C 线断开后，服务结束本次会话并恢复手机原有音频路由。

## 5. 原生 Android 服务需求

### 5.1 服务生命周期

- 应用使用 Kotlin 开发，核心能力运行在 Android 前台服务中。
- 服务应在首次配置后保持可用，并显示符合 Android 规范的常驻通知。
- 支持开机后恢复服务；不得依赖不合规的后台保活或隐藏进程方案。
- 服务应监听 USB 设备连接、断开、授权和音频设备变化事件。
- 服务被系统回收后，应在系统规则允许的范围内自动恢复。
- 应提供家属管理界面，用于首次授权、设备检测、豆包配置、故障诊断和停止服务。

### 5.2 自动触发流程

检测到目标 Newline 设备后，服务依次执行：

1. 读取 USB Vendor ID、Product ID、接口类型和音频能力。
2. 判断设备是否为已配置的 Newline NP-S2201。
3. 获取 USB 访问权限；首次连接需要用户确认，后续行为取决于手机系统是否记住授权。
4. 等待 Android 注册 USB 音频输入和输出设备。
5. 校验麦克风和扬声器均可用。
6. 保存手机当前音频路由。
7. 请求音频焦点并尝试选择 Newline 输入、输出设备。
8. 启动豆包。
9. 调用已经过同类程序验证的豆包启动方式，直接进入连续对话。
10. 持续监控 USB 连接、音频路由、豆包运行状态和异常。

### 5.3 豆包启动与控制

- 服务应通过 Android 标准 Intent 启动豆包，不使用模拟点击作为正式方案。
- “启动本应用后自动拉起豆包，并通过 Newline 进入连续对话”已有参考程序实际实现，因此作为本项目必须复现的基线能力。
- 开发时应对参考程序的运行环境、启动参数、Intent、系统配置、权限及音频路由行为进行对照记录。
- 在获得参考程序源码时，优先复用或移植其已验证的豆包启动协议；只有 APK 时，在取得程序权利人许可后进行行为分析和接口观测。
- 应将豆包包名、入口 Activity、Intent Action、URI、Extras 及版本兼容规则集中封装，避免散落在业务代码中。
- 豆包版本升级导致入口失效时，应用应识别失败状态、记录诊断信息并提供升级适配能力。

### 5.4 适老化要求

- 完成首次配置后，日常核心流程不得要求老年用户操作手机。
- 状态提示使用大字号、高对比度和简短中文语音提示。
- 成功连接时可播报“设备已连接，可以开始说话”。
- 连接失败时应给出可理解的提示，例如“连接失败，请重新插线”，避免显示技术错误码。
- 应提供家属模式，查看设备状态、授权状态、最近错误和测试麦克风/扬声器。
- 误拔线、应用被关闭或网络异常后，应尽量自动恢复；无法恢复时应提供明确提示。

## 6. 音频功能需求

### 6.1 设备连接与识别

- 支持 Android 16 手机，首批兼容设备包括主流 vivo 手机。
- 支持 Newline NP-S2201 会议终端。
- 支持通过 USB Type-C 有线连接传输双向音频。
- 手机应能够识别会议终端提供的 USB 音频输入和输出能力。
- 连接建立后应向用户显示连接成功、当前音频设备或异常状态。
- 重复插拔 Type-C 线后，系统应能够重新自动建立音频链路。

### 6.2 自动音频路由

- 连接成功后，豆包的录音输入自动使用 Newline NP-S2201 麦克风。
- 豆包的播放输出自动使用 Newline NP-S2201 扬声器。
- 输入和输出必须同时工作，满足双向语音交互。
- 连接期间来电、系统提示音或其他应用抢占音频焦点后，应尽可能自动恢复豆包的音频链路。
- Type-C 断开后不得继续占用已失效的 USB 音频设备。

### 6.3 拾音

- 支持会议室正常距离内的语音采集。
- 拾音音量应稳定，避免明显过小、削波、爆音或持续底噪。
- 应利用终端或系统具备的回声消除、噪声抑制和自动增益能力。
- 豆包播放回复时，应避免扬声器声音被麦克风重复采集并触发错误对话。
- 支持用户在豆包回复期间打断并继续提问；若豆包或设备不支持打断，应明确降级为回复结束后继续拾音。

### 6.4 音频播放

- 豆包的全部对话语音通过 Newline NP-S2201 扬声器播放。
- 播放声音应清晰、连续，不得出现明显卡顿、失真、重复或异常中断。
- 支持通过会议终端或手机调节播放音量。
- 播放结束后，系统应自动恢复拾音或保持可打断拾音状态。

### 6.5 连续对话

- 用户首次进入连续对话模式后，可连续完成多轮问答。
- 每一轮应自动完成“拾音—传输—豆包处理—播放—继续拾音”的状态切换。
- 非异常情况下，不要求用户每轮点击麦克风按钮。
- 应提供开始、暂停、继续和结束连续对话的控制入口。
- 长时间无语音时可进入等待状态；用户再次说话后应恢复对话。
- 网络异常、豆包无响应或音频设备异常时，应显示原因并支持重试。

## 7. 非功能需求

### 7.1 性能

- USB 音频链路建立时间目标不超过 5 秒。
- 用户开始说话后，音频应实时、持续传输，不出现可感知的周期性断流。
- 本地音频采集与播放链路新增延迟目标不超过 200 毫秒，不包含豆包云端处理时间。
- 连续完成 30 轮对话时，不应出现音频路由丢失、应用崩溃或必须重新插线的问题。

### 7.2 稳定性

- 支持至少连续运行 2 小时。
- 支持锁屏、解锁及豆包前后台切换后的合理恢复；受 Android 后台限制无法继续时，应向用户明确提示。
- 插拔、音频焦点变化和应用重启后，不得遗留麦克风占用。

### 7.3 隐私与安全

- 首次使用时，必须由用户授予麦克风及 USB 设备访问权限。
- 未开始连续对话或对话已经结束时，不得持续采集音频。
- 界面应明确显示当前是否正在拾音。
- 音频数据的采集、传输和存储应遵循豆包及 Android 平台的隐私策略。

## 8. 已知技术基线与 Android 16 约束

- Type-C 线必须支持数据传输，不能使用仅充电线。
- Newline NP-S2201 必须能够通过其 Type-C 接口向外提供标准 USB Audio Class 输入和输出能力；仅支持视频投屏、触控回传或充电的接口无法满足本需求。
- 手机与终端之间必须正确协商 USB Host/Device 角色、供电方向和 USB 音频功能。
- Android 16、手机厂商系统和豆包应用必须允许选择或跟随 USB 音频输入、输出设备。
- 已有参考 Android 程序可以在启动后自动拉起豆包，并通过 Newline 实现连续对话。本项目以相同手机、豆包版本和 Newline 环境作为首个兼容基线。
- USB 音频设备通常由系统统一管理，第三方应用不能保证在所有厂商系统上无提示、无授权地自动切换。
- 普通应用不能强制指定另一个应用的录音输入设备。服务可以监听和影响系统音频状态，但豆包最终使用哪个输入设备由 Android、厂商系统及豆包自身实现共同决定。
- Android 16 对后台拉起界面有限制。本项目需要复现参考程序采用的合规启动路径，并验证从 USB 物理连接事件触发时仍可自动拉起。
- Android 的 USB 权限、麦克风权限、通知权限和前台服务授权至少需要在首次配置时由用户或家属确认，无法合法做到安装后完全静默授权。
- 若 Newline 作为标准 USB Audio Class 设备被 Android 系统接管，服务不应自行读取原始 USB 音频端点，以免与系统或豆包争用设备。
- 开发前必须通过真机验证 NP-S2201 的 Type-C 端口能力及 vivo 目标机型的 USB 音频兼容性。

## 9. 异常处理

- 连接后未识别到 USB 音频设备：提示检查线缆、接口模式和设备能力。
- 只识别到输入或输出：提示当前不支持完整双向对话，并阻止进入完整连续对话模式。
- 麦克风权限被拒绝：说明授权方法并提供重新授权入口。
- USB 访问授权被拒绝：保持手机原音频路由，并允许再次授权。
- 连接中断：立即停止使用 USB 设备，恢复手机音频并提示连接已断开。
- 豆包或网络异常：停止当前轮等待，保留重试和结束对话入口。
- 出现严重回声或啸叫：降低播放音量、暂停播放或切换为半双工模式。

## 10. 验收标准

### 10.1 基本链路

- 使用指定 Type-C 数据线连接 vivo Android 16 测试手机和 Newline NP-S2201。
- 5 秒内识别到会议终端的麦克风和扬声器能力。
- 豆包录音输入来自 NP-S2201 麦克风。
- 豆包回复仅从 NP-S2201 扬声器正常播放。
- 断开线缆后，手机自动恢复原音频输入和输出。

### 10.2 自动化与连续对话

- 完成一次性配置后，连接 Type-C 线能够自动触发服务流程。
- 自动拉起豆包并进入连续对话，不要求用户点击豆包界面。
- 无需每轮点击按钮，连续完成至少 30 轮有效问答。
- 每轮回复结束后可以自动继续拾音。
- 支持打断时，用户在回复播放过程中说话可终止当前播放并发起下一轮；不支持时，系统按设计完成半双工切换。
- 全程无明显回声自激、重复触发、爆音、持续卡顿或音频路由丢失。

### 10.3 稳定性与恢复

- 连续运行 2 小时无崩溃和不可恢复的音频异常。
- 完成 20 次插拔测试，每次均能建立或恢复正确音频路由。
- 电话或其他应用短暂抢占音频后，返回豆包可继续对话或给出明确恢复操作。

## 11. 实施阶段

### 第一阶段：参考程序基线复现

- 获取 NP-S2201 真机、Type-C 数据线和目标 vivo Android 16 手机。
- 获取朋友已实现程序的 APK；条件允许时同时获取源码、构建配置和使用说明。
- 记录参考程序可正常运行的手机型号、Android 版本、豆包版本、Newline 固件、线缆及首次授权步骤。
- 在授权范围内采集参考程序的包信息、Manifest、启动日志、Intent、USB 枚举和音频设备变化日志。
- 验证 Newline 是否枚举为双向 USB Audio Class 设备。
- 验证系统及豆包是否实际使用 Newline 麦克风和扬声器。
- 确认参考程序拉起豆包连续对话的具体触发方式。
- 在本项目中先复现“打开应用即启动豆包连续对话”，再增加“插入 Newline 自动触发”。

第一阶段以参考程序的实际行为为验收基准，不再将自动拉起豆包连续对话视为未知可行性问题。

### 第二阶段：最小可用版本

- 实现首次配置和权限引导。
- 实现前台服务、USB 监听、设备识别、状态通知和异常恢复。
- 实现启动豆包及可用范围内的音频路由辅助。
- 实现适老化提示和家属诊断页面。

### 第三阶段：兼容性与稳定性

- 覆盖指定 vivo 机型及系统版本。
- 完成插拔、长时间运行、音频焦点抢占、锁屏和网络异常测试。
- 根据豆包官方接口能力完善全自动连续对话。

## 12. 待确认事项

- 朋友已实现程序的 APK、源码或可供测试的安装包。
- 参考程序的授权使用范围，是否允许复用源码、协议参数或进行 APK 行为分析。
- 参考程序已验证的手机型号、Android 版本、豆包版本和 Newline 固件版本。
- NP-S2201 的准确硬件版本、系统版本及 Type-C 接口规格。
- NP-S2201 是否原生支持 USB Audio Class 双向音频。
- 目标 vivo 手机的具体型号、系统版本和 USB 设置。
- 豆包应用版本，以及该版本是否支持 USB 外接麦克风、USB 音频输出和连续对话。
- 是否必须支持回复过程中的语音打断，即全双工交互。
- 是否允许安装独立 Android 应用，或必须完全依赖现有豆包应用。
- 产品是否需要覆盖 vivo 之外的其他 Android 16 品牌和机型。
